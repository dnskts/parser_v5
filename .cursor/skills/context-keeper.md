## При начале нового чата (новая сессия)

Если это начало новой сессии или контекст проекта неизвестен —
ПЕРЕД любой работой выполни следующее:

1. Прочитай файл `CURRENT_STAGE.md` в корне проекта ЦЕЛИКОМ
2. Прочитай файл `structure.md` в корне проекта
3. Подтверди пользователю что контекст загружен:

```
✅ Контекст проекта загружен из CURRENT_STAGE.md

Проект: [название и краткое описание]
Стек: [основные технологии]
Текущий статус: [что реализовано, что в процессе]
Известные проблемы: [перечислить кратко]
Последнее изменение: [дата и суть]

Готов к работе. Жду задачу.
```

4. НЕ ВЫДУМЫВАЙ информацию о проекте, которой нет в CURRENT_STAGE.md
5. Если CURRENT_STAGE.md не существует или пуст — сообщи об этом
   и предложи создать его, проанализировав проект
6. Если информации в CURRENT_STAGE.md недостаточно для выполнения
   задачи — сначала изучи конкретные файлы исходного кода,
   а затем ОБНОВИ CURRENT_STAGE.md недостающей информацией

## Когда активируется (обновление файла)

Этот скилл ДОЛЖЕН выполняться:
- После КАЖДОГО завершённого действия по запросу пользователя
- После КАЖДОГО изменения в структуре проекта, логике, конфигурации
- После КАЖДОГО создания, удаления или существенного изменения файла
- ПЕРЕД завершением ответа пользователю (как финальный шаг)

## Что делать

### Шаг 1: Обновить CURRENT_STAGE.md в корне проекта

Файл CURRENT_STAGE.md должен ВСЕГДА содержать следующие секции
в указанном порядке. Файл должен быть САМОДОСТАТОЧНЫМ — любая 
AI-модель, прочитав ТОЛЬКО этот файл, должна полностью понимать 
проект и мочь продолжить работу.

#### Обязательная структура CURRENT_STAGE.md:

```markdown
# [Название проекта] — Текущее состояние
**Последнее обновление:** [дата и время]
**Обновлено после:** [краткое описание последнего действия]

## 1. Общее описание проекта
[Что это за проект, его цель, основная функциональность.
Статистика: количество файлов, строк кода, парсеров.]

## 2. Технологический стек
[Языки, фреймворки, библиотеки, расширения, версии.]

## 3. Структура проекта
[Ключевые директории и файлы с назначением.
Ссылка на structure.md для полной структуры.]

## 4. Архитектура и взаимосвязи
[Pipeline обработки (ASCII-диаграмма), потоки данных Web UI,
ключевые архитектурные решения с обоснованиями,
детальная логика ключевых компонентов.]

## 5. Форматы данных
[Структура входных/выходных данных с примерами JSON,
таблицы обязательных полей, особенности форматов.]

## 6. Компоненты системы (парсеры, модули)
[Интерфейсы/контракты, описание каждого компонента,
входные данные, логика, вспомогательные методы.]

## 7. Внешние интеграции (API)
[Параметры подключения, формат запросов/ответов,
обработка ошибок, логирование.]

## 8. Web UI
[Каждая страница: что показывает, какие API использует,
особенности рендеринга (AJAX vs серверный).]

## 9. Текущее состояние разработки
[✅ Реализовано, 🔄 В процессе, 📋 Запланировано, ⚠️ Проблемы]

## 10. Последние изменения (краткий лог)
[Таблица последних 15 изменений. Старые → CHANGELOG_AI.md]

## 11. Важные соглашения и правила
[Именование, форматы, совместимость, стиль кода.]

## 12. Конфигурация и окружение
[Как запустить, settings.json, директории, права.]

## 13. Контекст для AI-ассистента
[Критические знания, quirks, предупреждения,
что нельзя менять, что нужно обновлять вместе.]
```



### Шаг 2: Обновить CHANGELOG_AI.md

При каждом обновлении CURRENT_STAGE.md добавить запись
в начало CHANGELOG_AI.md:

```markdown
## [Дата и время] — [Краткое описание]

**Запрос пользователя:** [суть того, что попросил пользователь]

### Что было сделано

### Изменённые файлы
- `[путь/файл]` — [что изменено]

### Принятые решения
- [Решение и почему]
```

### Шаг 3: Синхронизация зеркал `.txt`

Если в этой сессии изменялись **любые** из файлов: `SisPrompt.md`, `CURRENT_STAGE.md`, `CHANGELOG_AI.md`, `structure.md` — перед завершением ответа выполни в корне проекта:

`php scripts/sync-docs-to-txt.php`

Это обновляет одноимённые файлы `*.txt` (UTF-8, то же содержимое, что и у соответствующих `.md`). Нужно для сред, где удобнее plain text.

## Правила обновления

1. **НЕ УДАЛЯТЬ** информацию из CURRENT_STAGE.md — только обновлять
   и дополнять
2. **Секцию "Последние изменения"** — максимум 15 записей,
   старые переносить в CHANGELOG_AI.md
3. **Быть конкретным** — вместо "обновлён компонент" писать
   "в файле src/Header.tsx добавлен пропс isAuthenticated"
4. **Ссылки на файлы** — всегда указывать относительные пути
5. **Не выдумывать** — записывать только то, что реально сделано
6. **Архитектура** — при изменении взаимосвязей ОБЯЗАТЕЛЬНО
   обновить секцию 4
7. **Если CURRENT_STAGE.md не существует** — создать с нуля,
   проанализировав весь проект

## Правило глубины описания

- **АРХИТЕКТУРА и ВЗАИМОСВЯЗИ** — максимальная детализация
- **КОНКРЕТНЫЕ ИЗМЕНЕНИЯ** — средняя (что + зачем, без кода)
- **РУТИННЫЕ ОПЕРАЦИИ** (CSS, мелкие фиксы) — кратко в таблице

Если описание компонента требует больше 30 строк —
создать отдельный файл `docs/[компонент].md` и дать ссылку
из CURRENT_STAGE.md
