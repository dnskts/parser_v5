## 2. Рекомендуемые следующие шаги (с пояснениями)

1. **Открыть сетевой доступ к SFTP**  
   Сейчас сервер парсера не может достучаться до 10.4.175.11:22 (таймаут). Нужно, чтобы администратор сети открыл доступ с сервера, где запущен parser_v5, до SFTP-сервера поставщика. После этого синхронизация через «Запустить» и автообработку начнёт реально забирать XML.

2. **Обновлять SisPrompt и CURRENT_STAGE после любых изменений**  
   По правилам проекта после изменений обновляются CURRENT_STAGE.md, CHANGELOG_AI.md и при необходимости structure.md. SisPrompt.md — краткий системный промпт для веб-нейросетей; его стоит обновить, если изменились ключевые правила или структура (например, число фикстур, новые эндпоинты API). См. также скилл update-structure.

3. **При добавлении нового поставщика по API**  
   Следовать разделу 3 ниже: создать парсер, папку input, при необходимости — модуль приёма данных от API (если поставщик отдаёт заказы по API, а не файлами).

4. **Защита паролей в settings.json**  
   Вынести пароли API и SFTP из открытого файла: использовать переменные окружения (api_password, sftp_password) или отдельный secrets.json вне веб-корня. При чтении настроек подставлять значения из окружения, если заданы.

5. **Ограничение доступа к api.php**  
   Ввести проверку токена или HTTP Basic Auth для действий run, clear_logs, clear_json, чтобы предотвратить случайный или злонамеренный запуск снаружи.

6. **Поддержка нескольких SFTP-источников**  
   Сейчас одна секция sftp в settings. Для нескольких поставщиков с разным SFTP: массив sftp_sources с local_path на каждую папку input/; в process.php по очереди вызывать SftpSync для каждой записи.

7. **Мониторинг размера логов**  
   Настроить ротацию или алерт при превышении размера app.log, api_send.log, sftp_sync.log (например, уведомление при >50 МБ или ежедневная очистка старых записей).

8. **Алерты при ошибках SFTP**  
   При runSftpSync() при errors > 0 писать в отдельный лог или отправлять уведомление (почта, Telegram), чтобы оператор знал о сбоях загрузки.

9. **Тесты на ApiSender retry**  
   Добавить в test.php или отдельный скрипт проверку: при retry_attempts > 0 и имитации 5xx/таймаута убедиться, что выполняются повторные попытки и результат логируется.

10. **Документация формата cache_index.json**  
    Если будет введён кеш json/cache_index.json — описать в CURRENT_STAGE.md и structure.md структуру файла (поля, пример), чтобы разработчики и скрипты могли его использовать единообразно.

## 3. Подключение новых поставщиков через API (подробно)

Ниже описано, что значит «подключить нового поставщика» в этом проекте и как это делается — максимально простым языком.

### 3.1. Два варианта поступления данных от поставщика

В текущей архитектуре заказы от поставщика попадают в систему **в виде файлов** (XML в папках `input/...`). Реализовано два способа появления этих файлов:

1. **Ручная выгрузка**  
   Кто-то вручную кладёт XML-файлы в папку `input/имя_поставщика/`. Система при запуске обработки находит их и обрабатывает.

2. **SFTP-синхронизация**  
   Настроена для одного поставщика («Мой агент»): по расписанию или по кнопке «Запустить» система подключается к SFTP-серверу поставщика, скачивает новые XML в `input/moyagent/` и перемещает обработанные файлы на SFTP в папку Processed. То есть «получение файлов по сети» уже есть, но только для одного каталога и одного формата.

«Подключение поставщика через API» может означать два разных сценария:

- **Вариант A: Поставщик отдаёт заказы по HTTP API (JSON/XML по запросу или webhook).**  
  Тогда в проекте нужно добавить **модуль, который запрашивает данные у API поставщика** (или принимает webhook), сохраняет их в виде XML (или сразу преобразует в ORDER) и либо кладёт файлы в `input/имя_поставщика/`, либо передаёт данные в существующий конвейер (парсер → JSON → отправка в 1С). То есть «API» здесь — это способ **получить данные от поставщика**, а не только «отправить в 1С».

- **Вариант B: Поставщик по-прежнему отдаёт только файлы (например, по SFTP или по почте), но вы хотите добавить поддержку его формата.**  
  Тогда «подключение» — это только **новый парсер** и папка в `input/`. Никакого отдельного «API поставщика» в коде может не быть — достаточно парсера и, при необходимости, своей настройки SFTP (если у поставщика свой SFTP).

Ниже расписаны оба случая: как добавить поставщика с файловым форматом (общий случай) и как добавить поставщика, данные от которого приходят по HTTP API.

---

### 3.2. Что такое «парсер» в этом проекте

Парсер — это класс на PHP, который:

- «знает» папку поставщика в `input/` (например, `moyagent`, `demo_hotel`);
- умеет читать **конкретный формат XML** (или другого формата) этого поставщика;
- превращает один файл заказа в **единый формат ORDER** (массив с полями UID, INVOICE_NUMBER, INVOICE_DATA, CLIENT, PRODUCTS и т.д.), который уже умеет сохранять и отправлять в 1С текущая система.

То есть парсер не «подключается к API» сам по себе — он только **преобразует файл в ORDER**. Откуда файл взялся (ручная выгрузка, SFTP, или скачан по HTTP API) — решается отдельно.

### 3.3. Подключение нового поставщика с файловым форматом (без API поставщика)

Если у нового поставщика данные приходят **файлами** (XML в папку или по SFTP):

1. **Создать папку для файлов**  
   В каталоге `input/` создать папку с латинским именем без пробелов, например `input/new_supplier/`. Внутри создать подпапки `Processed/` и `Error/` (система создаст их сама при первом запуске, но можно создать сразу).

2. **Создать файл парсера**  
   В папке `parsers/` создать новый файл, например `parsers/NewSupplierParser.php`. В файле должен быть класс с именем, например, `NewSupplierParser`, реализующий интерфейс `ParserInterface` (файл `core/ParserInterface.php`).

3. **Реализовать три метода в парсере**  
   - `getSupplierFolder()` — вернуть строку с именем папки, например `'new_supplier'` (то же имя, что у папки в `input/`).  
   - `getSupplierName()` — вернуть человекочитаемое название, например `'Новый поставщик'` (для логов и интерфейса).  
   - `parse($xmlFilePath)` — принять путь к XML-файлу, прочитать файл, разобрать его и вернуть **один массив в формате ORDER**. Формат ORDER описан в CURRENT_STAGE.md (поля UID, INVOICE_NUMBER, INVOICE_DATA, CLIENT, PRODUCTS, внутри продукта — поля по спецификации RSTLS). При ошибке чтения или разбора нужно выбросить `Exception` с текстом ошибки.

4. **Подключить зависимости парсера**  
   В начале файла парсера указать:  
   `require_once __DIR__ . '/../core/ParserInterface.php';`  
   `require_once __DIR__ . '/../core/Utils.php';`  
   Для генерации UID использовать только `Utils::generateUUID()`.

5. **Ничего больше в коде не менять**  
   Класс `ParserManager` при каждом запуске сканирует все `*.php` в `parsers/`, находит классы, реализующие `ParserInterface`, и регистрирует их. Папка из `getSupplierFolder()` автоматически начнёт обрабатываться: система будет искать в ней `*.xml`, вызывать ваш парсер, сохранять JSON в `json/` и отправлять в 1С (если API включён).

6. **Если у поставщика свой SFTP**  
   Сейчас в настройках есть одна секция `sftp` в `config/settings.json`. Чтобы поддерживать несколько SFTP (по одному на поставщика), потребуется доработка: например, массив `sftp_sources`, где каждый элемент — конфиг с `local_path` = `input/new_supplier`, и в `process.php` или отдельном скрипте по очереди вызывать `SftpSync` для каждой записи. Это уже расширение текущей логики.

Итого для «просто нового формата файлов»: **новая папка в input/ + один новый файл парсера в parsers/** — и поставщик подключён.

### 3.4. Подключение поставщика, который отдаёт заказы по HTTP API

Здесь имеется в виду: **поставщик даёт заказы не файлами в папку, а по API** (вы вызываете его URL и получаете JSON или XML, или поставщик шлёт данные на ваш URL — webhook).

Что нужно сделать в проекте:

1. **Понять формат данных от API**  
   Получить от поставщика описание API: метод (GET/POST), URL, авторизация (ключ в заголовке, Basic Auth и т.д.), формат ответа (JSON или XML, структура полей). Примеры запросов и ответов очень помогут.

2. **Решить, где в проекте будет жить запрос к API**  
   Варианты:  
   - **Отдельный скрипт** (например, `fetch_supplier_api.php`), который по расписанию (cron) или по кнопке вызывает API поставщика, получает список заказов, для каждого заказа сохраняет ответ в файл в `input/new_supplier/order_123.xml` (или .json), после чего можно запустить обычную обработку (`process.php`). То есть «получение по API» только подготавливает файлы; дальше работает существующий конвейер (парсер читает файл из `input/new_supplier/`).  
   - **Встроить вызов в process.php**  
   Перед циклом по папкам вызвать функцию «забрать данные от поставщика X по API» и сохранить их в `input/new_supplier/` в виде файлов. Тогда при следующем шаге Processor подхватит эти файлы и обработает парсером.

3. **Реализовать клиент API поставщика**  
   В отдельном файле (например, `core/SupplierApiClient.php` или `fetchers/NewSupplierFetcher.php`) написать код, который:  
   - читает настройки (URL, ключ, логин/пароль) — лучше из `config/settings.json` (новая секция, например `suppliers_api.new_supplier`);  
   - делает HTTP-запрос к API поставщика (curl или file_get_contents с контекстом);  
   - разбирает ответ (JSON или XML);  
   - для каждого заказа в ответе сохраняет данные в файл в `input/new_supplier/`. Имя файла должно быть уникальным (например, номер заказа + расширение .xml или .json).  
   Если API возвращает уже один большой файл со всеми заказами — тогда либо сохранить его как один файл и парсер пусть разбирает много заказов из одного файла (текущие парсеры рассчитаны на один заказ на файл — придётся в парсере в цикле отдавать несколько ORDER или договориться, что один файл = один ORDER), либо на стороне fetcher-а разбить ответ на несколько файлов по одному заказу.

4. **Парсер под формат от API**  
   Если API возвращает JSON, а не XML — можно сделать парсер, который в `parse()` принимает путь к файлу: если расширение .json, читать JSON и преобразовывать в ORDER. Либо fetcher при получении JSON сразу сохраняет в виде XML в едином формате, а парсер остаётся XML. Главное — в итоге один вызов `parse()` для одного файла возвращает один массив ORDER (или массив из одного элемента, если договориться о таком формате и поправить Processor).

5. **Безопасность и ошибки**  
   Пароли и ключи API не хранить в коде — только в конфиге (или переменных окружения). Обрабатывать таймауты и коды ответа (4xx, 5xx): логировать, при необходимости повторять запрос (retry). Логи писать в `app.log` или отдельный лог, как для SFTP.

6. **Расписание**  
   Если забор по API делается отдельным скриптом — запускать его по cron перед или вместе с `process.php`. Если встроен в `process.php` — тогда при каждом «Запустить» и по cron будет сначала запрос к API поставщика, затем сохранение файлов, затем обычная обработка.

Кратко: **подключение поставщика по API = написать код, который получает данные от его API и кладёт их в виде файлов в `input/имя_поставщика/`, плюс парсер, который эти файлы превращает в ORDER.** Остальное (сохранение JSON, отправка в 1С) уже делает текущая система.

### 3.5. Чек-лист: новый поставщик (файлы или API)

- [ ] Определить, откуда приходят данные: файлы в папку, SFTP или HTTP API.
- [ ] Создать папку `input/имя_поставщика/` (и при необходимости Processed/, Error/).
- [ ] Создать класс парсера в `parsers/`, реализующий `ParserInterface` (getSupplierFolder, getSupplierName, parse).
- [ ] В `parse()` возвращать массив в формате ORDER (см. CURRENT_STAGE.md); при ошибке — throw Exception.
- [ ] Если данные приходят по API — добавить модуль/скрипт, который запрашивает API и сохраняет ответы в файлы в `input/имя_поставщика/`.
- [ ] Настройки API/SFTP поставщика хранить в `config/settings.json` (новые секции при необходимости).
- [ ] Проверить обработку через «Запустить» и просмотр результата на data.php и в логах API 1С.

После этого новый поставщик будет обрабатываться так же, как «Мой агент» и демо-отель: файлы в его папке → парсер → JSON → отправка в 1С.
